# Контрольні питання

## Лабораторна 8

### 1. У чому полягає принцип Documentation as Code?

Documentation as Code (DaC) — підхід, за яким документація зберігається в тому самому Git-репозиторії, що й код, пишеться у текстових форматах (Markdown), проходить code review та версіонується нарівні з програмним кодом. Це дозволяє застосовувати до документації ті самі інструменти: pull requests, CI/CD-пайплайни, автоматичну перевірку та публікацію.

### 2. Що таке Single Source of Truth?

Single Source of Truth (SSoT) — принцип, за яким кожен факт або вимога мають рівно одне авторитетне джерело. У цьому проєкті `SSD.md` є єдиним місцем, де визначаються функціональні вимоги. Усі інші документи *посилаються* на ідентифікатори з SSD, але не дублюють їх зміст. Це унеможливлює розбіжності між документами при оновленні вимог.

### 3. Чому Markdown є базовим форматом для документації?

Markdown — текстовий формат, який:

- читається без інструментів (plain text у будь-якому редакторі);
- зберігається в Git і підтримує diff / blame;
- конвертується в HTML, PDF, сайт (MkDocs, Hugo тощо) без зусиль;
- не потребує ліцензованого ПЗ (на відміну від Word).

### 4. Як забезпечується актуальність документації у DaC-підході?

У DaC-підході актуальність забезпечується організаційно та технічно:

- **Pull Request-процес:** зміна коду, що впливає на API або поведінку, не може бути злита без одночасного оновлення відповідного документа (перевіряється на code review).
- **CI/CD:** автоматичний пайплайн будує сайт документації при кожному merge в `main`; якщо конфігурація MkDocs містить помилку — білд падає.
- **Traceability Matrix:** пов'язує вимоги та тест-кейси, тому видалення або зміна вимоги одразу видно через відсутній TC-посилання.

## Лабораторна 9

### 1. Чому API-документація є контрактом?

Контракт — це угода між двома сторонами, яку обидві зобов'язані дотримуватись. `openapi.yaml` є саме таким: backend гарантує, що відповідає рівно тому формату, що описаний; frontend гарантує, що надсилає запити рівно в тому форматі, що описаний. Якщо один бік порушує контракт — CI pipeline ловить це через OpenAPI validation до того, як код потрапить у production.

### 2. У чому перевага API-first?

API-first дозволяє frontend і backend командам розробляти паралельно: frontend мокує відповіді на основі `openapi.yaml`, не чекаючи реалізації. Крім того, помилки в проектуванні API виявляються на етапі review специфікації (дешево), а не після написання коду обох сторін (дорого).

### 3. Яку роль відіграє Swagger UI?

Swagger UI перетворює `openapi.yaml` на інтерактивний портал, де розробник або QA може безпосередньо виконати запит до API, не відкриваючи Postman або curl. Це знижує поріг входу для нових членів команди та дозволяє тестувати API прямо з документації.

### 4. Навіщо потрібен Storybook?

Storybook вирішує для UI те саме, що Swagger UI — для API: дозволяє розробляти, тестувати та документувати компоненти в ізоляції від бізнес-логіки та backend. Дизайнер може переглянути всі стани компонента (loading, error, disabled) без розгортання повного застосунку.

### 5. Чому документація має бути частиною Definition of Done?

Документація, написана "потім", ніколи не пишеться. Якщо DoD не вимагає оновленої документації при кожній зміні коду — документація поступово застаріває і перестає відображати реальний стан системи. Включення документації в DoD робить її частиною нормального workflow, а не додатковим завданням, яке відкладається нескінченно.
