# Лабораторна робота №2

**Тема:** Практичне впровадження систем якості та безпеки

**Проект:** Smart-Gov (Система електронного голосування на базі блокчейн)

## Завдання 1. Оптимізація процесу за ISO 9001 (PDCA Workshop)

**Проблема:** "Кожен третій реліз містить критичний баг, який завалює продакшн середовище".

**1. Plan (Проектування змін):**
Для вирішення проблеми впроваджується інструмент **SonarQube** разом із вимогою **Mandatory Code Review & 80% Unit Test Coverage**.
*Впровадження:* Ми підключаємо SonarQube до CI/CD пайплайну. Налаштовуємо "Quality Gate", який буде автоматично "падати", якщо покриття нового коду тестами менше 80% або знайдено критичні вразливості чи баги. Додатково в системі контролю версій встановлюємо правило, що жоден код не може бути злитий (merged) без хоча б одного апруву від іншого розробника.

**2. Do (Впровадження):**
Як змінюється Git Flow команди:

1. Розробник більше не може робити коміти напряму в `main` чи `develop`. Він створює окрему гілку `feature/назва-задачі`.
2. Після завершення роботи розробник створює Pull Request (PR) у `develop`.
3. Автоматично запускається GitHub Actions (або інший CI), який компілює код, проганяє Unit-тести та відправляє звіт у SonarQube.
4. Якщо SonarQube показує, що покриття менше 80% або є критичні помилки, PR блокується (червоний хрестик). Розробник повинен доопрацювати код.
5. Коли CI зелений, мінімум один інший розробник (Senior) має провести Code Review, залишити коментарі та натиснути "Approve".
6. Тільки після успішного CI та апруву код можна злити в `develop`.

**3. Check (Контроль метрик):**
Протягом місяця будемо моніторити такі показники:

* **У Jira:** Кількість критичних багів, знайдених після релізу (має зменшитись до нуля або одиничних випадків).
* **У Jira:** "Cycle Time" або середній час від початку розробки задачі до її релізу (для контролю швидкості).
* **У GitHub Actions / SonarQube:** Відсоток успішних та неуспішних пайплайнів, динаміка Code Coverage та технічного боргу.

**4. Act (Коригування):**
*Якщо швидкість розробки впала вдвічі, а якість не зросла:*

* **Перегляд правил SonarQube:** Можливо, аналізатор дає забагато хибних спрацювань (false positives), через які розробники витрачають час на несуттєві правки. Вимкнемо другорядні стилістичні правила, залишивши лише блокування критичних вразливостей.
* **Адаптація порогу:** Тимчасово знизимо вимогу Coverage з 80% до 60% для старих модулів (legacy), залишивши 80% тільки для нового функціоналу.
* **Навчання команди:** Проведемо воркшоп з написання ефективних Unit-тестів, оскільки розробникам, ймовірно, не вистачає навичок швидкого написання тестів.

## Завдання 2. Практичний аудит безпеки за Тріадою CIA

| Сценарій (Що сталось?) | Параметр (C, I чи A) | Технічне рішення (Як запобігти автоматично?) |
| :--- | :---: | :--- |
| **Адміністратор БД випадково видалив таблицю з логами** | **Availability** (Доступність) | **Впровадження Point-in-Time Recovery (PITR)** для бази даних (напр. AWS RDS), що дозволяє відновити БД на будь-яку секунду в минулому. Також, логування варто перенести у **WORM-сховище** (Write Once Read Many, напр., AWS S3 Object Lock), щоб навіть адміністратор фізично не міг видалити або змінити існуючі логи голосування. Застосування IaC (Terraform) та заборона ручного доступу до production БД. |
| **Зловмисник підмінив номер гаманця (ідентифікатор) в API-запиті** | **Integrity** (Цілісність) | **Криптографічний підпис запитів (Request Signing) + mTLS**. Клієнтський додаток перед відправкою запиту підписує його тіло (payload) приватним ключем виборця. Сервер бекенду спочатку валідує криптографічний підпис. Якщо payload був змінений посередником або зловмисником, підпис не співпаде, і запит буде автоматично відхилено (HTTP 400/401). |
| **Стажер виклав приватний ключ від сервера у публічний GitHub** | **Confidentiality** (Конфіденційність) | **Впровадження Secret Scanning у CI/CD (напр. GitHub Advanced Security, git-secrets або TruffleHog).** Цей інструмент автоматично сканує кожен коміт (у вигляді pre-commit hook або перевірки під час PR) і блокує відправку коду, якщо знаходить формати ключів чи паролів. Для управління секретами слід використовувати **HashiCorp Vault** або **AWS Secret Manager**, щоб ключі взагалі не зберігалися у вигляді тексту в репозиторії, а інжектились як змінні оточення під час деплою. |

## Завдання 3. Проектування "Control List" (Annex A ISO 27001)

**Обраний компонент:** Модуль авторизації виборців (Voter Authentication Module)

1. **Контроль доступу:**
    * Використання суворої політики паролів (мінімум 12 символів, великі, малі, цифри, спецсимволи).
    * Обов'язкова **Багатофакторна автентифікація (MFA)**: для доступу до електронного бюлетеня виборець повинен підтвердити вхід через одноразовий код (TOTP із Google Authenticator/Authy) або використати Кваліфікований електронний підпис (наприклад, інтеграція з Дія.Підпис).
2. **Шифрування:**
    * **У спокої (Data at Rest):** Персональні дані виборців у базі даних шифруються за допомогою симетричного алгоритму **AES-256-GCM**.
    * **В русі (Data in Transit):** Весь трафік між пристроєм виборця та серверами Smart-Gov захищається протоколом **TLS 1.3** із забороною використання застарілих версій (TLS 1.1/1.2).
3. **Логування:**
    Приклад структури логу (JSON), який фіксує підозрілу активність (наприклад, серія невдалих спроб входу):

    ```json
    {
      "timestamp": "2026-06-19T10:15:30Z",
      "severity": "WARNING",
      "event_category": "AUTHENTICATION",
      "event_type": "FAILED_MFA_ATTEMPT",
      "voter_id": "voter-abc-12345",
      "source_ip": "198.51.100.14",
      "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)...",
      "details": {
        "reason": "Invalid TOTP code provided",
        "consecutive_failures": 4
      },
      "system_action": "Account temporarily locked for 30 minutes"
    }
    ```

## Завдання 4. Симуляція процедури сертифікаційного аудиту (IEEE 1012)

**Роль:** Внутрішній аудитор

**1. Checklist з 5 питань для перевірки процесу верифікації коду:**

1. Чи налаштовано блокування прямого "пушу" в основну гілку (main/master) з обов'язковою вимогою використання Pull/Merge Requests?
2. Чи проходить код автоматичну перевірку на наявність відомих вразливостей в залежностях (напр., npm audit, Snyk) перед злиттям?
3. Чи встановлений обов'язковий мінімальний поріг покриття коду автоматизованими тестами (Unit Test Coverage > 80%), нижче якого збирання вважається неуспішним?
4. Чи налаштовано обов'язковий статичний аналіз коду (SAST, напр. SonarQube) для виявлення "code smells" та порушень безпеки на етапі CI?
5. Чи вимагається проведення Code Review мінімум одним незалежним Senior-розробником перед тим, як код буде допущено до інтеграції?

**2. Докази (artifacts), які я вимагатиму від команди:**

1. **Скріншоти конфігурації гілок (Branch Protection Rules):** з налаштувань репозиторію (GitHub/GitLab), де чітко видно увімкнені галочки "Require a pull request before merging" та заборону прямого пушу.
2. **Логи перевірки залежностей (Dependency Scan Logs):** вивантажений звіт від інструментів типу `npm audit` або Snyk з CI/CD пайплайну, що підтверджує відсутність критичних вразливостей у залежностях.
3. **Звіти про покриття коду тестами (Test Coverage Reports):** згенерований звіт з CI/CD (наприклад, Jacoco/Istanbul), який підтверджує досягнення порогу покриття понад 80%.
4. **Звіти зі статичного аналізу (SAST Reports):** вивантажений PDF-звіт або посилання на дашборд SonarQube, де підтверджується проходження Quality Gate (відсутність критичних "code smells" та вразливостей).
5. **Приклади схвалених PR (Pull Requests):** посилання на 2-3 нещодавні Pull Requests, де наявні докази проведеного рев'ю (коментарі, обговорення) та фінальний "Approve" від Senior-розробника перед "Merge".
