# Лабораторна робота №7

**Тема:** Проектування архітектури програмної документації та системи забезпечення якості  
**Варіант системи:** Мікросервіс автентифікації користувачів (REST API)

---

## 1. Аналіз системи та цільової аудиторії документації

### Опис системи

Мікросервіс автентифікації — це незалежний програмний компонент, призначений для безпечного керування обліковими записами користувачів та їх доступом до інших сервісів системи. Головна мета мікросервісу: реєстрація нових користувачів, перевірка їхніх облікових даних (логін/пароль) та генерація JWT-токенів для подальшої безпечної взаємодії з іншими модулями.
Система не має власного графічного інтерфейсу (UI) для кінцевих користувачів. Взаємодія з мікросервісом відбувається виключно через **REST API**. Основними споживачами (типом користувачів) цього API є інші мікросервіси системи (наприклад, сервіс замовлень) та клієнтські додатки (Web/Mobile), які відправляють запити від імені реальних користувачів.

### Цільова аудиторія документації

Оскільки продукт є інфраструктурним (API), цільова аудиторія документації є суто технічною:

* **Backend / Frontend розробники:** для розуміння форматів запитів/відповідей (API контракти) та інтеграції мікросервісу у свої додатки.
* **QA-інженери (тестувальники):** для створення тестових сценаріїв, автоматизації API-тестів та перевірки безпеки.
* **DevOps-інженери:** для налаштування CI/CD, розуміння вимог до середовища, бази даних та розгортання в Kubernetes/Docker.
* **Security-спеціалісти:** для аудиту алгоритмів хешування паролів та механізмів роботи з токенами.

## 3. Проектування SSD (System Specification Document)

### 3.1. Призначення системи та межі (Scope)

**Межі системи:** Мікросервіс відповідає *тільки* за створення облікового запису, перевірку паролів, генерацію, валідацію та відкликання токенів доступу (JWT).
**Поза межами системи:** Мікросервіс *не зберігає* розширені профілі користувачів (аватарки, адреси, історію покупок) та не виконує бізнес-логіку інших систем.

### 3.2. Основні функції (Функціональні вимоги)

1. **FR-01: Реєстрація користувача.** Система повинна приймати email та пароль, хешувати пароль та створювати новий запис у базі даних.
2. **FR-02: Авторизація (Вхід).** Система повинна перевіряти email та пароль і у разі успіху повертати Access Token та Refresh Token.
3. **FR-03: Валідація та оновлення токена.** Система повинна перевіряти дійсність Access Token та надавати новий за допомогою дійсного Refresh Token.

### 3.3. Обмеження та припущення

* Паролі ніколи не зберігаються у відкритому вигляді (використовується алгоритм `bcrypt`).
* Взаємодія з мікросервісом можлива виключно по захищеному протоколу HTTPS.
* Email користувача є унікальним ідентифікатором (не може бути двох акаунтів з одним email).

### 3.4. Нефункціональні вимоги

* **Продуктивність:** Система повинна обробляти не менше 1000 запитів на авторизацію за хвилину. Середній час відповіді на API-запит не повинен перевищувати 150 мс.
* **Доступність (Availability):** Система є критичним компонентом, тому її доступність повинна складати не менше 99.99% (High Availability).
* **Безпека:** Система повинна мати захист від Brute-force атак (блокування IP після 5 невдалих спроб входу протягом 10 хвилин).

## 4. Проєктування SDD (Software Design Document)

### 4.1. Загальна архітектура

Система побудована за архітектурним стилем **мікросервісів**. Внутрішня архітектура самого сервісу базується на багатошаровому підході (Layered Architecture): Presentation Layer (Controllers), Business Logic Layer (Services), Data Access Layer (Repositories).

### 4.2. Перелік компонентів та їх взаємодія

* **Auth Controller:** Відповідає за прийом HTTP REST запитів, первинну валідацію вхідних даних (формат email, довжина пароля) та повернення HTTP відповідей.
* **Auth Service:** Містить основну бізнес-логіку (перевірка чи існує користувач, виклик сервісу хешування, прийняття рішень).
* **JWT Manager:** Утиліта для криптографічного створення та розшифровки JWT токенів.
* **User Repository:** Компонент, що абстрагує роботу з базою даних (виконання SQL-запитів).

**Схема взаємодії:** Клієнт -> `Auth Controller` -> `Auth Service` -> `User Repository` -> Database. Після успішної перевірки `Auth Service` звертається до `JWT Manager` для генерації токена і повертає його через контролер клієнту.

### 4.3. Короткий опис API інтерфейсів

* `POST /api/v1/auth/register` — приймає JSON з email та password, повертає `201 Created`.
* `POST /api/v1/auth/login` — приймає JSON з email та password, повертає `200 OK` та об'єкт з токенами `{"accessToken": "...", "refreshToken": "..."}`.
* `POST /api/v1/auth/refresh` — приймає `refreshToken`, повертає новий `accessToken`.

## 5. Проектування ISD (Infrastructure Specification Document)

### 5.1. Середовище розгортання

Мікросервіс розгортається у хмарному середовищі (наприклад, AWS або Google Cloud) з використанням контейнеризації **Docker**. Управління контейнерами здійснюється за допомогою оркестратора **Kubernetes**.

### 5.2. Основні інфраструктурні компоненти

* **API Gateway:** Єдина точка входу, яка маршрутизує запити до мікросервісу автентифікації та знімає SSL-сертифікати.
* **App Containers:** Кілька реплік (екземплярів) мікросервісу автентифікації для забезпечення відмовостійкості.
* **PostgreSQL (Relational DB):** Використовується для надійного зберігання облікових даних користувачів.
* **Redis (In-memory Cache):** Використовується для збереження "чорного списку" (blacklist) відкликаних токенів та лімітування запитів (Rate Limiting).

### 5.3. Принципи масштабування

Використовується **горизонтальне масштабування**. Kubernetes Horizontal Pod Autoscaler (HPA) автоматично збільшує кількість контейнерів мікросервісу, якщо середнє навантаження на CPU перевищує 70% або кількість запитів різко зростає. База даних PostgreSQL масштабується за принципом Primary-Replica (один сервер для запису, декілька для читання).

## 6. Документація якості та простежуваність

### 6.1. Test Strategy (Стратегія тестування)

* **Рівні тестування:**
  * *Unit-тести:* тестування бізнес-логіки та `JWT Manager` (розробники).
  * *Integration / API тести:* автоматизована перевірка REST-ендпоінтів за допомогою Postman/Newman (QA).
  * *Performance тести:* перевірка роботи під навантаженням за допомогою JMeter (чи витримає сервіс 1000 req/min).
* **Підхід:** API-first тестування. UI-тестування не застосовується через відсутність графічного інтерфейсу. Автотести запускаються автоматично в CI/CD пайплайні при кожному Pull Request.

### 6.2. Матриця простежуваності (Traceability Matrix)

Ця матриця гарантує, що кожна вимога з розділу SSD покрита відповідними тест-кейсами, що унеможливлює випуск неперевіреного функціоналу.

| Вимога (SSD ID) | Опис вимоги | Тест-кейс (TC ID) | Тип тесту | Очікуваний результат |
| :--- | :--- | :--- | :--- | :--- |
| **FR-01** | Реєстрація користувача | **TC-01.1** | API | Створення юзера, статус `201 Created` |
| **FR-01** | Реєстрація користувача | **TC-01.2** | API (Negative)| Помилка `409 Conflict` (email вже існує) |
| **FR-02** | Авторизація (Вхід) | **TC-02.1** | API | Успішний вхід, отримання валідного JWT |
| **FR-02** | Авторизація (Вхід) | **TC-02.2** | API (Negative)| Помилка `401 Unauthorized` (невірний пароль)|
| **FR-03** | Валідація / Refresh | **TC-03.1** | API | Видача нового токена за валідним refresh |

## Додаток: Відповіді на контрольні питання

1. **Чим SSD відрізняється від SDD?**
    * **SSD** (System Spec. Document) описує *ЩО* система повинна робити (бізнес-вимоги, функції, користувачі).
    * **SDD** (Software Design Document) описує *ЯК* система це робить (архітектура, код, бази даних, патерни).
2. **Яка роль ISD у життєвому циклі ПЗ?**
    * ISD визначає фізичне та віртуальне середовище, де буде "жити" код (сервери, мережі, Docker, хмари). Без ISD DevOps-інженери не зможуть правильно розгорнути продукт, забезпечити його масштабування та безпеку на рівні серверів.
3. **Навіщо потрібна матриця простежуваності?**
    * Для перевірки того, що жодна вимога клієнта не залишилася без тестування (coverage), і навпаки — що тестувальники не витрачають час на тестування функцій, яких немає у вимогах. Вона поєднує бізнес (вимоги) та якість (тести).
4. **Чому документація впливає на якість продукту?**
    * Документація усуває двозначність: розробники чітко знають, що писати, QA — що тестувати. Це зменшує кількість багів на етапі проектування (Shift-left testing), прискорює онбординг нових працівників і дозволяє системно контролювати стабільність продукту.
